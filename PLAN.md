# Introduction

The EU-project's [DIREKTION Assessment Screening Framework](https://www.direktion-network.org/dasf) (DASF) toolset consists of several core tools designed to simplify and systematize the assessment and screening process for disaster management. In addition to a user guide to support you through the assessment process. The toolset analysis is aligned with the latest EU priorities for disaster management, including the EU civil security taxonomy.

The DASF has four steps.

1. Initiation and Preparation (I&P)
   In this phase, stakeholders are brought together to define the scope of the disaster management assessment. This step includes identifying the relevant disaster types, such as natural hazards (e.g., floods or wildfires) or man-made risks (e.g., cyberattacks or industrial accidents). A clear understanding of the disaster context is crucial for successful assessments in subsequent phases.
2. Needs and Gaps Assessment (NGA)
   This phase focuses on identifying the capability gaps that hinder effective disaster response. The Needs and Gaps Assessment tool enables practitioners to assess their current capabilities and compare them against pre-defined disaster management standards. This process ensures that all critical gaps are identified, allowing for targeted interventions.
   For example, during a flood scenario, a capability gap might be identified in the interoperability of communication systems between emergency services in neighboring countries. The NGA tool helps visualize these gaps, providing a basis for further evaluation in the solution assessment phase.
3. Solution Assessment (SA)
   In the Solution Assessment phase, the identified gaps are addressed by evaluating potential technological and operational solutions. This assessment covers various aspects such as the technology readiness level (TRL), feasibility, and potential impact of the solutions.
   For instance, a flood response team might assess an early warning system developed under a Horizon 2020 project. The SA tool provides a systematic way to evaluate the maturity and applicability of the solution, ensuring that only the most promising innovations are prioritized for implementation.
4. Roadmapping (RM)
   The Roadmapping phase connects the results of the needs and gaps assessment and the solution assessment to long-term planning and policy development. This process supports EU institutions in aligning their future research and funding programs with the most critical disaster resilience needs. The roadmap helps ensure that solutions are not only identified but also scaled and sustained across multiple disaster contexts and regions.

## Goal

Create an implementation plan for a Single Page Application (SPA) that will implement the DASF-toolset.

## Use cases

### Single person

A single person should be able to follow the steps in the framework to determine the needs and gaps in a (Crisis Management) organization, finding solutions, and putting them on a roadmap.

### Workshop/group session

A facilitator should be able to prepare the workshops in the tool, present it to an audience, and invite the audience to join him/her in filling out the forms in the toolset.

### Roles

There are two roles: users and editors/facilitators, and there is no authentication required - you can switch freely between roles. Users can fill-out forms, editors can change questions, edit the framework, etc. Editing the framework involves enabling or disabling steps (or substeps), editing the list of hazard types, editing questionnaires, etc. Not adding more steps. Potentially, it may be interesting to separate users into stakeholders/end-users who are working in the disaster management domain, and solution providers, that provide applications and solutions addressing their gaps.

## Architecture

A SPA implemented in typescript + mithril.js using meiosis for state management (see the skill). The design components are from mithril-materialized (see the skill), the SPA should be multilingual using translate.js (see the skill). Forms should be created using mithril-ui-form, a library that converts JSON to HTML forms in mithril. There shall be no backend, and the application will be hosted via GitHub Pages. Local storage will be done via IndexedDB. Collaborative sessions should be using WebRTC, where the facilitator can present, and the workshop attendants can fill out forms. The current application already is based on this architecture, but we need to expand it. Existing pages in src/components can be replaced or updated, as required.

Also, the user should be able to select the GUI language and light or dark theme.
- Main page: After opening an existing session, or joining an external one, the main page is shown. It displays the steps of the framework and its sub-steps, and the current status of each step/sub-step (completed, partial, open). When creating a new session, it will by default start with the 4 steps in the DASF, but the facilitator can disable certain steps . The main page should also display a summary (based on the availability, of course): what hazard type(s) are considered, which capability gaps are identified/selected, which solutions are considered per gap, and when they are to be implemented. ## Collaboration

A facilitator can select which steps of the framework to do. For each step, he can edit the questions and the answer options, for example, see the src/components/settings-page.ts. He can also create a room in order to start a collaborative session. When users join the room, depending on their type (solution provider or not), they can either add new solutions, or answer questions. New solutions will be shared with everyone. User answers to questions will be shared too, and the average, maximum or minimum score will be shown. When end users add remarks, they are shared too. The facilitator can merge and summarize them, while keeping the original comments intact.
